# Known Bugs

Bugs noticed during testing that aren't immediate blockers. They get fixed when Step priorities allow.

| ID | Noticed | Severity | Component | Description |
|---|---|---|---|---|
| BUG-001 | 2026-05-06 | Low | Search | Search field doesn't handle Swedish characters (å, ä, ö). Typing without diacritics doesn't match stops whose names contain them, and vice versa. Need locale-aware normalisation (e.g. `Collator` / NFD + diacritic strip) so `"angerm"` matches `"Ångermanland"`. |
| BUG-002 | 2026-05-06 | **Fixed 2026-05-07 (runtime confirmed)** | Map / markers | When the map is zoomed out, the "400 nearest to center" markers all clump at the center while the rest of the visible map has no markers — looks unnatural. Possible fix: only render markers above a zoom threshold (e.g. zoom ≥ 12), or switch to true clustering (osmdroid-bonuspack `RadiusMarkerClusterer`) that shows summary icons when zoomed out and individual stops when zoomed in. **Fix shipped 2026-05-07:** chose the zoom-threshold path (lower complexity than full clustering). `MIN_STOP_ZOOM = 14.0` constant in `MainActivity`; `rebuildVisibleMarkers` skips marker construction below this level. Initial map zoom is 13, so user must zoom in once before stops appear — also addresses the "too distracting on first launch" UX feedback that drove this fix. |
| BUG-003 | 2026-05-06 | **Fixed 2026-05-07 (runtime confirmed)** | Map / markers | Default OSMDroid stop marker icon is too large when zoomed out — markers visually overlap and obscure the map tiles. Use a smaller custom icon (small dot / drawable) and possibly scale with zoom level. **Fix shipped 2026-05-07:** stop markers now use a 6 dp filled circle (`STOP_DOT_DIAMETER_DP = 6f`, `STOP_DOT_COLOR = 0xFF607D8B` muted blue-grey) instead of the default OSMDroid pin. `makeStopDot` was extended to accept a size parameter so commute-overlay stops keep the existing 12 dp size. Anchor changed from `ANCHOR_BOTTOM` to `ANCHOR_CENTER` to match the symmetric dot. |
| BUG-004 | 2026-05-07 | Pre-distribution blocker | Map / tile provider | App currently uses OSMDroid's default `TileSourceFactory.MAPNIK` which fetches from `tile.openstreetmap.org`. The OpenStreetMap Foundation's tile-usage policy prohibits this for distributed apps regardless of commercial intent (volume-based, not status-based). Must switch to a third-party tile provider (CartoDB / MapTiler / Mapbox / OpenFreeMap / similar) before any wider distribution — including free/non-commercial release. Fine to leave for development phase. Tracks NFR9. |
| BUG-005 | 2026-05-07 | **Fixed 2026-05-07 (runtime confirmed)** | Step 4 / direction labels | The bundled `sl-lines.json` produced by `extractGtfs` had empty `headsign` strings on some directions because SL's GTFS feed leaves `trip_headsign` blank for those trips (notably bus 3 in both directions). Caused empty direction labels AND, surfaced in Step 5, prevented direction-correct filtering of live vehicles. **Fix shipped 2026-05-07:** in `extractGtfs`, when `trip_headsign` is blank, fall back to the name of the trip's final stop. Re-ran extractor, regenerated `sl-lines.json`. **Runtime confirmed 2026-05-07:** Logcat showed populated headsigns ('Motalavägen', 'Tengdahlsgatan') instead of empty strings; matchDirection picked correct direction once paired with BUG-009 fix. |
| BUG-006 | 2026-05-07 | Low | Performance / UI jank | Bottom sheet open / close and saving a commute trigger 1.4–1.8 s frame times on the emulator (Logcat "Davey!" warnings + one "Skipped 93 frames"). Caused by main-thread work — likely a combination of OSMDroid rebuilding 400 stop markers on map idle, bottom sheet inflating Material Design components synchronously, and the polyline + stop-dot overlays being added in one shot. Possible fixes: pre-inflate the bottom sheet on activity start, throttle marker rebuilds harder, batch-add overlays on a worker, profile with Android Studio's CPU/Memory profilers to confirm the dominant cost. User flagged as low priority — visual polish, not a blocker. |
| BUG-007 | 2026-05-07 | Low (feature) | Commute config / quota | Cap each saved commute's active time window to a maximum of 30 minutes. Enforced in the bottom-sheet config UI (start/end picker validation). Rationale: protects the Trafiklab Bronze monthly quota even if a user saves a wildly long window by mistake; also matches realistic commute durations. Defer until quota usage is actually observed in practice. |
| BUG-008 | 2026-05-07 | Low | Map / overlays | When a newly-saved commute's polyline overlaps a previous commute on the map, the new one isn't visually drawn (or appears identical to the existing one). Suspected causes: same color picked from `COMMUTE_PALETTE` (only 5 colours, cycle by index), z-order making the older line render on top, or the overlapping polyline literally being identical-looking under stroke alpha 180. Investigation needed: log polyline endpoints + colours when saving, render z-order, possibly bump newer overlays up the overlay stack. Reported during Step 5 testing. |
| BUG-009 | 2026-05-07 | **Fixed 2026-05-07 (runtime confirmed, three-version fix arc)** | Step 5 / direction matching | SL Transport API's `direction` string (e.g. "Sofia" — shown on the bus's destination sign at the user's stop) doesn't always match the GTFS final-stop name (e.g. "Motalavägen" — the literal end of the route). **Fix v1 (2026-05-07):** added `directionCode: Int?` to `CommuteConfig`, saved from `StopLineOption.directionCode`. matchDirection preferred direction_id match. **Fix v2 (2026-05-07):** runtime test showed SL uses 1-based codes (config had directionCode=2, GTFS direction_id is 0-based) — added `code-1` heuristic fallback. **Fix v3 (2026-05-07):** consulted Trafiklab OpenAPI spec on user request — confirmed `direction_code` is 0/1/2 with 0 = unidentified, and Trafiklab support explicitly states SL Transport and GTFS Realtime systems "don't share IDs" and recommends matching via trip_id and stop_times. Replaced heuristic as primary with **stop-sequence-aware matching**: added `stopName: String?` to `CommuteConfig`, saved from bottom sheet, matchDirection finds the GTFS direction where `config.direction` appears AFTER `config.stopName` in the stop ordering. **Runtime confirmed 2026-05-07:** with stopName='Tullgårdsparken' populated and config.direction='Sofia', matchDirection returned direction_id=1 (Tengdahlsgatan) — the documentation-aligned answer, which also happens to be what the heuristic gave us. Both algorithms agree. Three buses correctly tracked toward Sofia. |
| BUG-010 | 2026-05-07 | **Fixed 2026-05-07 (runtime confirmed)** | Step 6 / vehicle marker InfoWindow | Tapping a vehicle marker shows the InfoWindow popup, but it can't be dismissed by tapping the map (no close affordance). The default OSMDroid behaviour SHOULD close it on map tap, but apparently doesn't trigger reliably in our setup. **Fix shipped 2026-05-07:** added a `MapEventsOverlay` at index 0 in the map overlay stack whose `singleTapConfirmedHelper` calls `InfoWindow.closeAllInfoWindowsOn(map)`. Returns `false` so the event continues to other overlays — only fires when no marker was tapped. |
| BUG-011 | 2026-05-07 | **Fixed 2026-05-07 (runtime confirmed)** | Map / zoom controls | OSMDroid's built-in zoom +/- buttons auto-hide when not in use AND are anchored at the bottom-center of the map, where they overlap the live_status banner. **Fix shipped 2026-05-07:** disabled the built-in controller (`map.zoomController.setVisibility(NEVER)`) and added two custom Buttons (`@id/zoom_in`, `@id/zoom_out`) anchored bottom-end with a 72 dp bottom margin to clear the live_status banner. Always visible. Wired to `map.controller.zoomIn()` / `zoomOut()`. The live_status banner itself is now hidden when state is `Idle` (was previously a 24 dp empty bar at the bottom even before tracking started). |
| BUG-012 | 2026-05-07 | Low — investigate later | Step 6 / vehicle quality | Vehicle markers frequently render as `UNCERTAIN` (greyed, "uncertain" tag in title) even when the polling cycle has just succeeded. Suspected cause: SL's GTFS-RT feed populates `vehicle.timestamp` with the bus's last GPS report time, which can lag the feed's emit time by tens of seconds. Our `STALE_THRESHOLD_MS = 60_000` may be too tight for this feed. Need empirical observation: log the distribution of `nowMs - tsMs` over a few hours of normal SL service to pick a sensible threshold (likely 90–120 s), or weight LIVE-vs-UNCERTAIN by feed-emit time when available. Defer until Step 7 (disruptions) or Step 10 (polish) — doesn't affect tracking correctness, only the visual confidence indicator. |
| BUG-013 | 2026-05-08 | Low (feature / refinement) | Notifications / commute alerts | User-defined GPS-trigger waypoint for commute notifications. User picks a point along the route polyline (or drops a custom point near the route); when the live GPS position of the tracked vehicle passes that point in the user's travel direction, the app fires a one-shot push notification ("your bus has reached <waypoint>" / "leave now"). Lets the user calibrate "when to leave home" to their own walking time to the stop, rather than relying on a fixed `eta <= 3` heuristic. Filed instead of the LeaveNow notification suggested in the design handoff — user explicitly preferred a configurable, user-controlled waypoint over a fixed-distance trigger. Defer until Phase 1 widget + foreground service ship; this is a polish/refinement on top of Step 8 (uses the same foreground-service tick that updates the widget; the trigger evaluation is a per-poll geometric check on the tracked vehicle's lat/lon vs. the saved waypoint). Considered for Step 10 polish or Phase 2 if scope expands. |
